Harness Engineering — 설계 고뇌, 답, 최종 아키텍처

==================================================
1부. 고뇌 목록 (Questions & Dilemmas)
==================================================

[고뇌 1] deleteCompany — 뭘 삭제해야 하나?
---
처음엔 "모든 걸 삭제"로 설계되어 있었다.
그런데 생각해보니: 전부 지울 거면 그냥 폴더 자체를 날리면 되지,
굳이 명령어가 필요한가?
그리고 PRD, report, 소스코드(develope/)는 개발자가 공들여 만든 결과물이다.
이걸 AI가 지워버리면 안 된다.

→ deleteCompany가 존재해야 하는 진짜 이유가 뭔가?
  "개발이 끝나고 서버에 올리려는데, AI 운영 파일까지 올리긴 싫다."
  배포 전 정리 도구였던 것이다.


[고뇌 2] createCompany — 재개발 시 어떻게 처리하나?
---
deleteCompany로 AI 운영 파일만 지웠다.
나중에 유지보수나 기능 추가를 위해 다시 개발을 시작하면?
createCompany를 다시 실행해야 하는데,
이때 PRD.md, Coding_Rule.txt, report.md가 이미 존재한다.
이걸 덮어쓰면 기존 작업이 날아간다.

→ createCompany가 두 가지 상황을 구분해야 한다:
  1. 처음 시작 (fresh) → 모든 파일 생성
  2. 재개발 (re-init) → AI 운영 파일만 재생성, 기존 content 파일 유지


[고뇌 3] 병렬 처리 — Wave 방식이 맞나?
---
처음엔 Wave(파동) 방식으로 태스크를 묶어 병렬 처리했다.
Layer 1이 모두 끝나야 Layer 2 시작.
그런데 실제로는: Layer 2의 Task-A는 Layer 1의 Task-X만 기다리면 되는데,
Task-Y가 늦게 끝난다고 Task-A까지 기다려야 하는가?

Wave가 끝나길 기다리는 건 비효율이다.
선행이 완료된 순간 즉시 다음 태스크를 시작해야 한다.


[고뇌 4] Manager AI가 많아지면 생기는 문제
---
프로젝트 규모가 크면 Manager AI를 여러 개 spawn한다.
그런데 여러 Manager AI가 동시에 AI_list.txt를 읽고 쓰면?
→ race condition 발생 가능성.
→ 서로 다른 Manager AI가 같은 파일을 수정하는 태스크를 맡으면?
→ 코드 파일 충돌.

Manager AI를 어떻게 나눠야 충돌이 없는가?
어떤 기준으로 나눠야 하는가?


[고뇌 5] Boss AI가 계속 블로킹되는 문제
---
Boss AI가 프로젝트 전체를 오케스트레이션하면,
프로젝트가 끝날 때까지 Boss AI는 계속 일하고 있다.
그 도중에 유저가 명령어를 입력하면?
→ Boss AI가 응답하지 못한다.

그렇다고 Boss AI를 중간에 끊으면 프로젝트 진행이 멈춘다.
어떻게 해야 유저가 도중에 개입할 수 있는가?


[고뇌 6] Sub AI — 재사용할까, 삭제할까?
---
Sub AI가 태스크를 완료했다.
다음 태스크가 생겼을 때 기존 Sub AI를 재사용할까, 새로 만들까?

재사용의 장점: 생성/삭제 오버헤드 없음.
삭제 후 새 생성의 장점: 이전 컨텍스트가 없는 깨끗한 상태. 토큰 효율 good.
삭제의 단점: 생성/삭제 시간 소모.

→ 실제로 Sub AI 삭제는 카운터 하나 줄이는 것 (단순 파일 I/O).
  새 Sub AI 생성도 spawn 명령 한 번.
  오버헤드가 거의 없다.
  그리고 이전 태스크 컨텍스트를 안고 가는 게 오히려 토큰 낭비.


[고뇌 7] Secretary AI — 필요한가?
---
Boss AI가 세그먼트 조율로 바쁘면,
유저 응대를 Secretary AI에게 맡기자는 아이디어가 있었다.

그런데: Boss AI는 세그먼트를 Manager AI에게 넘기고 나면
다음 세그먼트가 완료될 때까지 실질적으로 대기 상태다.
대기 중인 Boss AI가 유저를 직접 응대하면 되는데,
굳이 Secretary AI를 별도로 두어야 하는가?

→ 불필요. Boss AI가 세그먼트 간 대기 시간에 직접 응대.


[고뇌 8] 세그먼트 분할 기준 — 어떻게 나눠야 파일 충돌이 없나?
---
여러 Manager AI가 동시에 작업한다.
A Manager AI와 B Manager AI가 같은 소스 파일을 건드리면 충돌.
"어떤 기준으로 세그먼트를 나눠야 파일 충돌이 원천 차단되는가?"

Task 수로 나누면? → 같은 파일을 건드리는 Task가 쪼개질 수 있다.
도메인으로 나누면? → backend 안에서도 파일 충돌 가능.
레이어로 나누면? → 같은 레이어에 같은 파일 수정 Task가 있을 수 있다.

→ 가장 근본적인 기준: "같은 파일을 수정하는 Task는 무조건 같은 세그먼트."
  File Ownership이 최우선 분할 기준.


[고뇌 9] AI_list.txt 충돌 — 복수 Manager AI가 같은 파일을 쓰면?
---
소스 파일 충돌은 세그먼트 분할로 해결했다.
그런데 AI_list.txt 자체는 어떻게?
Manager AI-01과 Manager AI-02가 동시에 AI_list.txt에 항목을 추가/삭제하면?
→ 파일 write race condition.

초기 해결안: 세그먼트별 전용 섹션 → 각 Manager AI가 자신의 섹션만 수정.
그런데 생각해보니 더 근본적인 질문이 생겼다:
"AI_list.txt에 개별 AI 이름, STATUS, SPECIALTY를 기록할 필요가 있나?
Sub AI는 어차피 1회용이고, 작업 기록은 report/fragment/에 남는데."

→ AI_list.txt의 정체성 재정의 필요.
  명단 파일이 아닌 용량 추적 파일로 바꾸면
  충돌 문제 자체가 단순해진다.


[고뇌 10] Collector AI 실행 빈도 — 매 태스크마다 실행이 최선인가?
---
태스크가 완료될 때마다 Collector AI를 spawn해서 report.md를 업데이트했다.
그런데 규모가 큰 프로젝트에서는 Collector AI가 너무 자주 뜬다.
반대로 프로젝트가 끝나고 나서야 한 번 돌리고 싶은 경우도 있다.

→ 실행 시점을 유저가 선택할 수 있어야 한다.
  세그먼트 완료마다 (기본) / 프로젝트 완료 시 / 태스크마다 중 선택.


[고뇌 11] teamLimit — 무엇을 제한해야 하나?
---
기존: teamLimit = Sub AI 전체 최대 수 (단일 숫자).
그런데 실제로 필요한 제한은 두 가지다:
  1. 동시에 몇 개의 세그먼트(Manager AI)를 병렬 실행할 것인가?
  2. 세그먼트 하나 안에서 Sub AI를 몇 명까지 동시에 돌릴 것인가?

전자는 Boss AI 레벨의 병렬도, 후자는 Manager AI 레벨의 병렬도.
하나의 숫자로 이 두 가지를 동시에 제어하는 건 표현력이 부족하다.

2부. 답 (Solutions)
==================================================

[답 1] deleteCompany 재정의
---
정체성: 배포 전 정리 도구.

삭제 대상: AI가 운영하며 만든 임시/운영 파일만.
  - doc/AI_list.txt
  - doc/AI_anomaly.txt
  - doc/company_state.json
  - report/fragment/* (개별 보고서 조각)
  - add/registry.json, add/installed/*, add/config/*, add/runtime/*

보존 대상: 개발자의 지식과 결과물.
  - doc/PRD.md
  - doc/Coding_Rule.txt
  - develope/ (소스 코드)
  - report/report.md (최종 보고서)

추가 기능: 삭제 직전 PRD + report.md를 바탕으로 README.md 자동 생성 (선택).
기존 README.md 존재 시: 덮어쓰기 또는 README_프로젝트명.md로 분리 저장.


[답 2] createCompany — 두 모드 구분
---
실행 시 PRD.md, Coding_Rule.txt, report.md 존재 여부 감지.

Fresh (처음 시작):
  → 폴더 구조 전체 생성 + 모든 파일 생성

Re-init (재개발):
  → 폴더가 없으면 생성, AI 운영 파일만 재생성
  → PRD.md, Coding_Rule.txt, report.md는 건드리지 않음
  → "기존 파일 유지됨. 변경 사항은 /complainProject 또는 /newTask를 사용하세요."


[답 3] Wave → DAG 기반 즉시 실행
---
Wave 방식 폐기. Dependency 기반 즉시 실행으로 전환.

각 태스크는 자신의 deps(선행 태스크 목록)만 체크한다.
deps가 모두 완료된 순간 즉시 spawn. 레이어 전체를 기다리지 않는다.

Boss AI가 DAG 전체를 가지고 있으며,
세그먼트 완료 보고를 받을 때마다 새로 unlock된 세그먼트를 즉시 시작.


[답 4] 소스 파일 충돌 원천 차단 — File Ownership
---
세그먼트 분할 알고리즘:

1단계 DAG 생성: 태스크 → 노드, 의존성 → 방향 엣지.
2단계 Topological Layering: BFS로 레이어 구분.
3단계 Domain Clustering: 레이어 내 backend/frontend/infra/testing 클러스터.
4단계 File Ownership 적용:
  - 같은 파일을 수정하는 태스크 → 무조건 같은 세그먼트. (최우선 규칙)
  - Shared infra (DB schema, config, 공유 타입) → 별도 세그먼트.
  - Verification/Testing → 별도 세그먼트.

결과: Manager AI마다 전담 file_ownership이 배정되고, 경계 밖 파일은 절대 수정 불가.
→ Manager AI 간 소스 파일 충돌 구조적으로 불가능.


[답 5] Boss AI 블로킹 해소 — 세그먼트 위임 구조
---
Boss AI의 역할을 "태스크 실행"에서 "세그먼트 조율"로 격상.

Boss AI가 하는 일:
  1. DAG 작성 (전체 그림)
  2. 세그먼트 분할 + Dependency Contract 정의
  3. Manager AI에게 세그먼트 위임
  4. 완료 보고 수신 → 다음 세그먼트 unlock

Manager AI가 세그먼트를 받은 뒤부터는 완전 자율:
  - Sub AI spawn/관리
  - 태스크 순서 조율
  - Collector/Monitoring spawn

Boss AI는 세그먼트 간 대기 시간에 유저 응대 가능.
중간 개입이 필요하면 /interceptBoss로 다음 세그먼트 시작 보류.


[답 6] Sub AI 완료 후 즉시 삭제
---
태스크 완료 → fragment 보고서 작성 → Manager AI에게 보고 → 카운터 감소.
다음 태스크는 새 Sub AI가 담당.

이유:
  - 이전 컨텍스트 없는 깨끗한 상태 → 토큰 효율
  - "삭제" = 카운터 -1 (파일 I/O 최소)
  - 재사용이 오히려 불필요한 context 누적


[답 7] Secretary AI 폐기
---
Boss AI가 세그먼트 간 대기 중에 직접 유저 응대.
별도 AI 역할 불필요. 아키텍처 단순화.


[답 8] Dependency Contract
---
세그먼트마다 계약서를 붙인다:
  imports: 이 세그먼트가 실행되기 전에 다른 세그먼트에서 받아야 하는 결과물
  exports: 이 세그먼트가 완료 시 제공하는 결과물

imports가 없으면 즉시 시작 가능.
exports가 완료되면 그것을 imports로 가진 세그먼트를 unlock.

Boss AI가 exports/imports 계약을 추적하며 세그먼트 실행 순서를 결정.


[답 9] AI_list.txt 재정의 — 명단 파일 → 용량 추적 파일
---
개별 AI 이름/STATUS/SPECIALTY 기록 폐기.
이유: Sub AI는 1회용이고, 작업 기록은 report/fragment/에 이미 남는다.
     명단을 유지하는 건 중복 기록이고, 동시 write 충돌의 원인이었다.

새 구조:
  [Capacity]
  Manager Limit     : 3     ← Boss AI가 동시 운영 가능한 Manager AI 수
  Sub AI per Manager: 4     ← Manager AI 1개당 동시 운영 가능한 Sub AI 수

  [Status]
  Active Managers   : 2
    backend_core (Manager AI-01) : 2 Sub AI   ← Manager AI-01만 이 줄 수정
    frontend_ui  (Manager AI-02) : 1 Sub AI   ← Manager AI-02만 이 줄 수정
  Available Managers: 1

쓰기 권한 분리:
  - Boss AI: Active Managers 카운트 + 세그먼트 항목 추가/삭제 + Available 업데이트
             (순차적 작업 → 충돌 없음)
  - Manager AI-N: 자신의 세그먼트 항목 Sub AI 카운트만 수정
                  (각자 다른 줄 → 물리적 충돌 불가능)


[답 10] collectorMode — Collector AI 실행 시점 선택
---
/collectorMode [n] 명령어로 설정.

  1 (기본): Manager AI가 세그먼트 완료 시 1회 실행
            → 중간 진행 상황이 주기적으로 기록됨
  2: 프로젝트 전체 완료 시 Boss AI가 1회 실행
     → 보고서 작업을 최소화하고 싶을 때
  3: Sub AI 태스크 완료마다 실행
     → 실시간 보고서가 필요할 때


[답 11] teamLimit 분리 — managerLimit + subAILimit
---
단일 teamLimit → 두 개의 독립 설정으로 분리.

  managerLimit: Boss AI 레벨 병렬도 (동시 세그먼트 수)
  subAILimit:   Manager AI 레벨 병렬도 (세그먼트당 Sub AI 수)

/teamLimit 실행 시 대화형으로 두 값을 순서대로 입력받는다.
최대 동시 Sub AI 수 = managerLimit × subAILimit.


==================================================
3부. 최종 AI 아키텍처
==================================================

계층 구조
----------
[유저]
  ↓ 명령어/요청
[Boss AI]  ← Main AI, 항상 활성
  ↓ 세그먼트 위임
[Manager AI-N]  ← on-demand, 세그먼트당 1개, 완료 후 종료
  ↓ 태스크 위임
[Sub AI]  ← on-demand, 태스크당 spawn, 완료 후 즉시 소멸
  ↓ 완료 보고
[Collector AI] + [Monitoring AI]  ← collectorMode/monitoringEnabled 기준으로 spawn


Boss AI
--------
역할: 유저 소통 전담 + 전체 의존성 조율자
  - 유저 입력 수신 및 응답
  - PRD 분석, 전체 DAG 작성
  - 세그먼트 분할 (4단계 알고리즘)
  - Manager AI에게 세그먼트 + Dependency Contract 위임
  - exports/imports 계약 추적, 세그먼트 실행 순서 결정
  - 세그먼트 완료 보고 수신 → 다음 세그먼트 unlock
  - collectorMode=2 시: 프로젝트 완료 후 Collector AI 1회 spawn
AI_list.txt 쓰기:
  - Active Managers 카운트 + 세그먼트 항목 추가/삭제 + Available 업데이트
  - (Manager AI spawn 시 +1, 완료 시 -1)


Manager AI (Bounded Context Coordinator)
-----------------------------------------
역할: 세그먼트 경계 내 완전 자율 코디네이터
  - Boss AI로부터 Dependency Island + Dependency Contract 수령
  - 세그먼트 내 내부 DAG 분석, Sub AI 배치
  - Sub AI spawn 전: 자신의 항목 Sub AI 카운트 +1 (subAILimit 초과 시 보류)
  - Sub AI 완료 후: 자신의 항목 Sub AI 카운트 -1 → 보류 중 태스크 즉시 시작
  - 태스크 완료마다: collectorMode/monitoringEnabled 기준으로 spawn
  - 세그먼트 완료 시: exports 결과물 목록과 함께 Boss AI에게 보고 후 종료
AI_list.txt 쓰기:
  - 자신의 세그먼트 항목 Sub AI 카운트만 수정. 타 항목 절대 수정 불가.
file_ownership:
  - 배정된 경로 내에서만 코드 작성. 외부 파일 수정 절대 금지.
인스턴스 수: Active Managers ≤ managerLimit


Monitoring AI (on-demand)
--------------------------
역할: 자원 소모 감시자
  - 태스크 완료마다 Manager AI가 spawn (monitoringEnabled=true 시만)
  - fragment 보고서의 토큰 소모량, 오류 반복, 작업 범위 초과 점검
  - 이상 감지 시: doc/AI_anomaly.txt 기록


Collector AI (on-demand)
--------------------------
역할: 보고서 취합기
  - fragment 보고서를 읽어 report/report.md에 반영
  - 실행 시점은 collectorMode 설정에 따름:
      1 (기본): Manager AI가 세그먼트 완료 시 spawn
      2: Boss AI가 프로젝트 완료 시 spawn
      3: Manager AI가 태스크 완료마다 spawn


Sub AI (on-demand)
-------------------
역할: 실제 코드 작성자
  - Manager AI가 직접 spawn (동시 수 ≤ subAILimit)
  - 작업 전: 기존 코드 먼저 읽고 변경 범위 파악
  - file_ownership 경로에만 코드 작성
  - 완료 시: report/fragment/TASK-XXX_[ai-name].md 보고서 작성
  - Manager AI에게 완료 보고 → 카운터 감소 후 소멸
예시 역할 (고정 목록 아님):
  - Backend AI: 서버 로직, API, DB, 인증
  - Frontend AI: UI/UX, HTML/CSS/JS, 프론트 프레임워크
  - Debugger AI: 코드 검수, Coding Rule 준수 확인


AI_list.txt 구조
-----------------
[Capacity]
Manager Limit     : 3      ← /teamLimit으로 설정
Sub AI per Manager: 4      ← /teamLimit으로 설정

[Status]
Active Managers   : 2
  backend_core (Manager AI-01) : 2 Sub AI
  frontend_ui  (Manager AI-02) : 1 Sub AI
Available Managers: 1


세그먼트 분할 알고리즘 (Boss AI 수행)
--------------------------------------
STEP 1 — DAG 생성
  각 태스크: { id, description, deps, domain, files }
  taskCounter를 이어서 TASK-XXX 번호 부여

STEP 2 — Topological Layering (BFS)
  Layer 1: deps 없는 태스크 (루트 노드)
  Layer N: 모든 선행이 Layer N-1 이하인 태스크

STEP 3 — Domain Clustering
  레이어 내에서 backend / frontend / infra / testing 도메인 클러스터

STEP 4 — Segment 생성 + Dependency Contract
  최우선 규칙: 같은 파일 수정 태스크 → 무조건 같은 세그먼트
  Shared infra (DB schema, config, 공유 타입) → 별도 세그먼트
  Verification/Testing → 별도 세그먼트
  각 세그먼트: { segment_id, tasks, file_ownership, imports, exports }


태스크 완료 흐름
-----------------
Sub AI 완료
  → fragment 보고서 작성 (완료 시각, 작업 요약, 수정 파일, 토큰 소모량)
  → Manager AI에게 완료 보고
  → Manager AI: 자신의 항목 Sub AI 카운트 -1
  → Manager AI: 태스크 완료 처리
       ├─ collectorMode=3 → Collector AI spawn → report.md 반영
       └─ monitoringEnabled=true → Monitoring AI spawn → 이상 시 anomaly.txt 기록
  → Manager AI: 세그먼트 내 의존성 확인 → 새로 실행 가능한 태스크 즉시 spawn
  → 세그먼트 내 모든 태스크 완료 시:
       ├─ collectorMode=1 → Collector AI spawn → report.md 반영 → Boss AI에게 보고
       ├─ collectorMode=2 → 즉시 Boss AI에게 보고
       └─ collectorMode=3 → 즉시 Boss AI에게 보고 (이미 반영됨)
  → Boss AI: AI_list.txt에서 해당 세그먼트 항목 삭제, Available +1
  → Boss AI: exports를 imports로 가진 세그먼트 unlock → 즉시 Manager AI spawn
  → 모든 세그먼트 완료 시:
       └─ collectorMode=2 → Boss AI가 Collector AI 1회 spawn → 최종 report.md


권한 분리 요약
---------------
AI_list.txt 구조 (항목 추가/삭제)  → Boss AI 전담
AI_list.txt Sub AI 카운트          → 각 Manager AI가 자신의 항목만
report.md 수정                     → Collector AI 전담
Sub AI 생성                        → Manager AI 전담
의존성 그래프                      → Boss AI (전체) / Manager AI (세그먼트 내부만)
소스 파일 접근                     → 각 Manager AI는 자신의 file_ownership 경로만


핵심 설계 원칙
---------------
1. File Ownership이 세그먼트 분할의 최우선 기준.
   같은 파일 = 같은 세그먼트. Manager AI 간 소스 파일 충돌 구조적으로 불가능.

2. Boss AI는 조율자, Manager AI는 자율 실행자.
   Boss AI는 세그먼트 경계와 순서만 관리.
   세그먼트 내부는 Manager AI가 완전 자율.

3. Sub AI는 1회용.
   태스크 완료 즉시 소멸. 깨끗한 컨텍스트 = 토큰 효율.

4. Dependency Contract로 세그먼트 간 인터페이스 명시.
   imports/exports 계약이 세그먼트 실행 순서를 결정.

5. AI_list.txt는 명단이 아닌 용량 추적기.
   개별 AI 이름 불필요. 카운터만으로 충돌 없는 동시성 제어.

6. 병렬도는 두 레벨로 독립 설정.
   managerLimit (Boss AI 레벨) × subAILimit (Manager AI 레벨).

7. Collector AI 실행 시점은 유저가 선택.
   세그먼트 완료(기본) / 프로젝트 완료 / 태스크마다 중 선택.
